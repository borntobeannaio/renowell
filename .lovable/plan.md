
## Поддержка регулярных (повторяющихся) событий из ICS

### Проблема

Парсер ICS извлекает из каждого `VEVENT` только одну дату `DTSTART`/`DTEND`. Если событие содержит `RRULE` (например, `RRULE:FREQ=WEEKLY;BYDAY=MO`), все повторения игнорируются — в базу попадает только первое вхождение.

### Решение

Добавить в Edge-функцию `sync-ics-calendar` развёртку правил повторения (RRULE) в конкретные даты. Каждое вхождение станет отдельной записью в `calendar_events` с уникальным `external_uid`.

### Что изменится

#### 1. Edge-функция `sync-ics-calendar/index.ts`

- Добавить извлечение полей `RRULE`, `EXDATE` (исключённые даты) из каждого `VEVENT`.
- Реализовать функцию `expandRRule(dtstart, dtend, rrule, exdates)`, которая разбирает правило повторения и генерирует конкретные даты вхождений.
- Поддерживаемые параметры RRULE:
  - `FREQ` — DAILY, WEEKLY, MONTHLY, YEARLY
  - `INTERVAL` — интервал повторения (по умолчанию 1)
  - `COUNT` — максимальное число повторений
  - `UNTIL` — дата окончания повторений
  - `BYDAY` — дни недели для WEEKLY (MO, TU, WE, TH, FR, SA, SU)
- Горизонт генерации — 6 месяцев вперёд от текущей даты (чтобы не создавать бесконечное количество записей).
- Каждое вхождение получит `external_uid` = `{original_uid}__{YYYYMMDD}` для уникальности в базе.
- Исходное событие без RRULE продолжит работать как раньше (одна запись).

#### 2. Обработка EXDATE

- `EXDATE` содержит даты, исключённые из правила (отменённые встречи).
- При генерации вхождений такие даты будут пропущены.

### Пример

Событие в ICS:
```text
BEGIN:VEVENT
UID:abc123
SUMMARY:Планёрка
DTSTART:20260216T100000Z
DTEND:20260216T110000Z
RRULE:FREQ=WEEKLY;BYDAY=MO;COUNT=10
EXDATE:20260223T100000Z
END:VEVENT
```

Результат — 9 записей в базе (10 повторений минус 1 исключение):
- `abc123__20260216` — 16 февраля
- `abc123__20260302` — 2 марта
- `abc123__20260309` — 9 марта
- и т.д.

### Технические детали

- Все изменения в одном файле: `supabase/functions/sync-ics-calendar/index.ts`.
- Интерфейс `VEvent` будет расширен полями `rrule?: string` и `exdates?: string[]`.
- Функция `expandRRule` — чистая функция без внешних зависимостей, работает с Date-объектами.
- Лимит: максимум 200 вхождений на одно регулярное событие (защита от бесконечных правил без COUNT/UNTIL).
- Никаких изменений в базе данных не нужно — структура `calendar_events` подходит как есть.
